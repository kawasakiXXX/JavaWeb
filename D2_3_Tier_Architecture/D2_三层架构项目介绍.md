# D2 三层架构 Web 项目介绍

## 项目概述

这是一个经典的 **用户管理系统 REST API**，严格遵循三层架构（3-Tier Architecture）设计模式。提供用户的增删改查（CRUD）功能，数据暂存在内存中。

- **框架**: Spring Boot 4.1.0 + Spring Web MVC
- **JDK**: Java 17
- **构建工具**: Maven
- **内嵌服务器**: Tomcat (无需额外安装)
- **数据存储**: ConcurrentHashMap (模拟数据库，零外部依赖)

---

## 三层架构设计

整个 D2 项目的核心价值在于**架构演示**——它展示了企业级 Java Web 开发中最基础的职责分离模式：

```
┌─────────────────────────────────────────────────┐
│                  请求/响应边界                     │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │         Controller (控制层)              │    │
│  │  · 接收HTTP请求，解析参数                 │    │
│  │  · 调用Service，获取结果                  │    │
│  │  · 封装ResponseEntity返回给客户端         │    │
│  │  · 不包含任何业务逻辑                     │    │
│  └──────────────┬──────────────────────────┘    │
│                 │ 调用                           │
│  ┌──────────────▼──────────────────────────┐    │
│  │         Service (业务逻辑层)             │    │
│  │  · 校验数据合法性（非空检查等）            │    │
│  │  · 处理业务规则                           │    │
│  │  · 协调多个DAO完成复杂操作                │    │
│  │  · 不关心数据怎么存的                     │    │
│  └──────────────┬──────────────────────────┘    │
│                 │ 调用                           │
│  ┌──────────────▼──────────────────────────┐    │
│  │          DAO (数据访问层)                │    │
│  │  · 直接操作数据存储                      │    │
│  │  · 隐藏存储细节（Map/JDBC/MyBatis）       │    │
│  │  · 提供纯数据操作：增删改查               │    │
│  │  · 不关心数据怎么来的                     │    │
│  └──────────────┬──────────────────────────┘    │
│                 │ 读写                           │
│  ┌──────────────▼──────────────────────────┐    │
│  │    ConcurrentHashMap<Integer, User>     │    │
│  │         (模拟数据库)                      │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

### 为什么每层都用接口+实现？

| 层 | 接口 | 实现 | 好处 |
|----|------|------|------|
| DAO | `UserDao` | `UserDaoImpl` | 换存储（如从 Map 换成 MySQL）只需换实现类，Service 零改动 |
| Service | `UserService` | `UserServiceImpl` | 换业务规则不影响 Controller |
| Controller | 直接写（REST 接口天然是契约） | — | 无 |

## 🔄 一个完整请求的调用链

以 `POST /api/users` 新增用户为例：

```
Step 1. 客户端发送: POST http://localhost:8080/api/users
                    Body: {"name":"Alice","email":"alice@example.com"}

Step 2. Tomcat 接收请求 → DispatcherServlet 路由 → UserController.createUser()

Step 3. Spring 用 Jackson 把 JSON → User 对象

Step 4. Controller: 不做任何处理，直接调用 userService.registerUser(user)

Step 5. Service: 校验 name 和 email 非空 ✅
                 调用 userDao.save(user)

Step 6. DAO: idGenerator.getAndIncrement() → 1
              user.setId(1)
              userDB.put(1, user)
              返回带 ID 的 user

Step 7. Service: 返回 user 给 Controller

Step 8. Controller: ResponseEntity.ok(user)
                     Spring 用 Jackson 把 User → JSON

Step 9. 客户端收到: {"id":1,"name":"Alice","email":"alice@example.com"}
```

## 📡 API 接口文档

| 方法 | URL | 功能 | 请求体 | 响应 |
|------|-----|------|--------|------|
| `GET` | `/api/users` | 查询所有用户 | — | `[{...}, ...]` |
| `GET` | `/api/users/{id}` | 按ID查询 | — | `{...}` 或 404 |
| `POST` | `/api/users` | 新增用户 | `{"name":"...","email":"..."}` | 带自增ID的用户 |
| `PUT` | `/api/users/{id}` | 更新用户 | `{"name":"...","email":"..."}` | 更新后的用户 |
| `DELETE` | `/api/users/{id}` | 删除用户 | — | `"删除成功: id=X"` |

```
=== 1. 新增用户 Alice ===
```

## 📦 逐层代码解读

### Model 层 — `User.java`

```java
public class User {
    private Integer id;      // 用户ID（自增主键）
    private String name;     // 用户名
    private String email;    // 邮箱
}
```

纯数据载体（POJO）。`id` 不在构造时传入，由 DAO 的 `idGenerator` 自动分配。Spring MVC 用 Jackson 自动完成 JSON ↔ User 对象的序列化/反序列化。

---

### DAO 层（数据访问层） — `UserDao.java` / `UserDaoImpl.java`

```java
@Repository
public class UserDaoImpl implements UserDao {

    private final Map<Integer, User> userDB = new ConcurrentHashMap<>();  // 模拟数据库表
    private final AtomicInteger idGenerator = new AtomicInteger(1);       // 自增主键
```

核心设计要点：

- **`ConcurrentHashMap`** — 线程安全。真实项目里 Tomcat 每个请求一个线程，用普通 `HashMap` 在并发写入时会出问题。
- **`AtomicInteger`** — 线程安全的自增计数器。`getAndIncrement()` 等价于数据库的 `AUTO_INCREMENT`。
- **`save(User)`** — 自动分配 ID，然后把对象存入 Map。**返回保存后的对象**，这样 Controller 就能拿到带 ID 的完整数据返回给客户端。
- **`update(User)`** — 先检查 ID 是否在 Map 中存在，不存在返回 `null`。**存在性检查是 DAO 的职责**，不应丢给上层。
- **`deleteById(Integer)`** — `Map.remove()` 返回被删除的值，利用这个判断是否删除成功。

```java
public interface UserDao {
    List<User> findAll();             // 查询所有用户
    User findById(Integer id);        // 根据ID查询用户
    User save(User user);             // 新增用户，返回含ID的用户
    User update(User user);           // 更新用户
    boolean deleteById(Integer id);   // 根据ID删除，返回是否成功
}
```

### Service 层（业务逻辑层） — `UserService.java` / `UserServiceImpl.java`

```java
@Service
public class UserServiceImpl implements UserService {

    private final UserDao userDao;  // 构造器注入

    public UserServiceImpl(UserDao userDao) {
        this.userDao = userDao;
    }
```

**构造器注入 vs @Autowired 字段注入**：构造器注入是 Spring 官方推荐的最佳实践——

- 不可变（`final`），不会被意外修改
- 测试时可以手动 `new UserServiceImpl(mockDao)`
- 启动时就能发现依赖缺失（而不是运行时 NPE）

**业务校验逻辑**：

```java
public User registerUser(User user) {
    if (user.getName() == null || user.getName().isBlank()) {
        throw new IllegalArgumentException("用户名不能为空");
    }
    if (user.getEmail() == null || user.getEmail().isBlank()) {
        throw new IllegalArgumentException("邮箱不能为空");
    }
    return userDao.save(user);
}
```

- 校验放在 Service 层而不是 Controller——因为**业务规则属于业务层**，即使未来换了入口（比如从消息队列接收数据），校验逻辑不变。
- 用 `isBlank()` 而不是 `isEmpty()`——会同时拦截 `""` 和 `"   "`（纯空格）。

**更新逻辑**：

```java
public User updateUser(User user) {
    User updated = userDao.update(user);
    if (updated == null) {
        throw new IllegalArgumentException("用户不存在: id=" + user.getId());
    }
    return updated;
}
```

DAO 返回 `null` → Service 转换成明确的异常抛出 → 客户端知道为什么失败。

```java
public interface UserService {
    List<User> getAllUsers();         // 获取所有用户
    User getUserById(Integer id);     // 根据ID获取用户
    User registerUser(User user);     // 注册新用户（含业务校验）
    User updateUser(User user);       // 更新用户信息
    boolean deleteUser(Integer id);   // 删除用户
}
```

### Controller 层（控制层） — `UserController.java`

```java
@RestController              // = @Controller + @ResponseBody（每个方法自动返回 JSON）
@RequestMapping("/api/users")  // 统一前缀
public class UserController {

    private final UserService userService;

    public UserController(UserService userService) {
        this.userService = userService;
    }
```

Spring 注解映射表：

| 注解 | HTTP 方法 | URL 模式 | 对应操作 |
|------|-----------|----------|----------|
| `@GetMapping` | GET | `/api/users` | 查询所有 |
| `@GetMapping("/{id}")` | GET | `/api/users/1` | 按 ID 查询 |
| `@PostMapping` | POST | `/api/users` | 新增 |
| `@PutMapping("/{id}")` | PUT | `/api/users/1` | 更新 |
| `@DeleteMapping("/{id}")` | DELETE | `/api/users/1` | 删除 |

**`@PathVariable` vs `@RequestBody`**：

- `@PathVariable Integer id` → 从 URL 路径中提取
- `@RequestBody User user` → 从 HTTP 请求体 JSON 反序列化

**404 处理**：

```java
@GetMapping("/{id}")
public ResponseEntity<User> getUserById(@PathVariable Integer id) {
    User user = userService.getUserById(id);
    if (user == null) {
        return ResponseEntity.notFound().build();  // HTTP 404
    }
    return ResponseEntity.ok(user);  // HTTP 200 + JSON
}
```

用 `ResponseEntity` 精确控制 HTTP 状态码。

## 🔧 Spring 的依赖注入如何工作

Spring 启动时扫描 `@SpringBootApplication` 所在包及其子包：

```
JavaWebApplication.java  (@SpringBootApplication, 扫描起点)
├── D2/
│   ├── controller/
│   │   └── UserController    ← 发现 @RestController，创建 Bean
│   ├── service/
│   │   └── impl/
│   │       └── UserServiceImpl ← 发现 @Service，创建 Bean
│   └── dao/
│       └── impl/
│           └── UserDaoImpl     ← 发现 @Repository，创建 Bean
```

Spring 按照依赖关系自动装配：

1. 先创建 `UserDaoImpl`（无依赖）
2. 再创建 `UserServiceImpl`（注入 `UserDaoImpl`）
3. 最后创建 `UserController`（注入 `UserServiceImpl`）

如果哪一环缺失（比如忘了写 `@Repository`），启动时就会报错，不会等到运行才发现。

## ⚠️ 注意事项

- **数据不持久化**：数据存在 `ConcurrentHashMap` 中，重启服务即清空。这是刻意为之——保持项目零外部依赖，专注于三层架构演示。真实项目可以替换 `UserDaoImpl` 接入 MySQL 等数据库。
- **中文编码**：Windows 下使用 `curl` 时中文可能出现 UTF-8 编码问题。建议用 Postman 或浏览器测试中文数据，或使用英文数据。
- **线程安全**：`ConcurrentHashMap` + `AtomicInteger` 保证了并发安全，但仅限单机内存。真实项目需依赖数据库的事务机制保证一致性。
- **无认证授权**：当前 API 无任何安全控制，生产环境需加入 Spring Security 或 JWT 等认证机制。
